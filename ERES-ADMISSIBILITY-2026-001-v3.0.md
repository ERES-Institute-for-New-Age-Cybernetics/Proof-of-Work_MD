# **VIII.  Repeatable Drift: The Named Failure Mode**

Repeatable drift: stable output behavior produced by unstable or unverifiable internal transition logic. Three structural reasons ERES treats this as primary:

* Most likely to survive casual inspection. Consistent outputs are what evaluators check first. An engine optimized for consistent outputs while running on unstable internals passes surface-level evaluation indefinitely.

* It accumulates. Each consistent output adds weight to the trust position. When the underlying transition logic eventually produces an inconsistent output, accumulated trust amplifies the damage.

* MIEVM is specifically designed to surface it. Cross-instrument convergence at ≥8.5 across Claude, DeepSeek, Grok, and ChatGPT is not a quality metric — it is a drift detector. An engine running on unstable transition logic will not achieve convergent validation across four independent frontier instruments simultaneously. The ensemble exposes what single-instrument evaluation conceals.

The Register II answer to repeatable drift is byte-normative determinism via EAAP v1.5: state is measured by BERA from outside the engine, not inferred from engine self-report. An engine that reports its state has been trusted, not measured.
