# **5\. The Consolidation Pipeline**

The Consolidation Phase comprises eight stages, executed in order. Each stage has defined inputs, operations, and outputs. The pipeline runs sequentially against a growing Pie; the order of document interrogation matters and is specified in §5.3.

## **5.1 Stage 0 — Intake**

Input: the raw \~2GB corpus as delivered. Operation: recursive unzip of all archives (handling nested archives), format detection, file-type classification. Output: a flat staging directory plus an intake manifest recording original path, extracted path, size, content hash (SHA-256), and modification timestamp for every file.

## **5.2 Stage 1 — Inventory and Normalize**

Input: the staging directory from Stage 0\. Operation: extract plaintext content and metadata from every document (PDF via pypdf, DOCX via python-docx, MD/TXT direct, images via OCR, code files as text). Compute fuzzy hash (ssdeep or MinHash) per document. Extract embedded ERES-IDs, author strings, and content dates. Output: normalized document records ready for interrogation.

## **5.3 Stage 2 — Interrogation Ordering**

Interrogation runs sequentially against a growing Pie. The order in which documents are interrogated affects slice-admission outcomes. Canonical priority order:

8. Locked whitepapers bearing version-lock strings and ERES-IDs.

9. MIEVM-validated documents from the March–April 2026 production sprint.

10. Documents bearing RG\#\#\#\# publication identifiers.

11. Documents bearing any ERES-ID.

12. All remaining documents, ordered by content date (earliest first).

## **5.4 Stage 3 — Interrogation**

For each normalized document, the Interrogation Runner invokes the BRAINS interrogation protocol. The question the protocol asks is: what unique point(s) does this document contribute that are not already in the Pie? The runner receives back zero-or-more candidate slices. Each candidate carries: proposed HowWay expression, proposed KeyNoteCommentary handle, proposed EDF₁/EDF₂/EDF₃ scores, and origin document pointer.

## **5.5 Stage 4 — MIEVM Validation**

Each candidate slice is routed to all four MIEVM nodes (Claude, Grok, DeepSeek, ChatGPT) for independent scoring. Each node evaluates the candidate across all three EDF axes and returns scores plus rationale. Consensus rules:

* Four-node agreement: auto-admit.

* Three-of-four agreement: admit with minority-flag recorded.

* Two-two split: route to the Author for manual adjudication.

* Unanimous rejection: reject the candidate; the document may still contribute via other candidates.

## **5.6 Stage 5 — Uniqueness Check and Admission**

Candidates that clear MIEVM validation are checked against the existing Pie for uniqueness. A candidate is unique if no existing slice carries the same HowWay expression semantic content. Outcomes:

* Unique: admit as new slice, assign slice-ID, record origin document.

* Duplicate: do not admit; add origin document as corroborator on the existing matching slice.

* Near-duplicate with version-lock advantage: supersede the existing slice, move the superseded slice to historical layer, record supersession.

## **5.7 Stage 6 — S$ID and Anchor Linkage**

Every admitted slice is linked to: (a) one or more S$IDs in the SELF-$ELF Ledger identifying whose personal-equals-$ is at stake, and (b) one or more entries in the @begins Anchor Registry identifying which beginning the slice addresses. A slice with no S$ID or no anchor linkage is flagged for Author review — it may indicate the slice is insufficiently grounded in GAIA’s governance function.

## **5.8 Stage 7 — Codex Projection**

After each admission (or in scheduled batches), the Codex Projector regenerates the three Codex views over the current Pie. Each Codex surfaces slices whose dominant EDF score matches its axis, ordered by chapter-shell skeleton (defined by BRAINS), with cross-references to slices appearing in secondary Codices. Codices are living documents; manuscript deliveries to MIT Press are version-locked snapshots.

## **5.9 Stage 8 — Archive Ledger Update**

Every document that yielded no unique slice is recorded in the Archive Ledger with its corroboration trail: this document fully corroborates slices X, Y, Z; contributed no new slice; processed on date D. The Archive Ledger proves the corpus was fully processed, not selectively mined.
