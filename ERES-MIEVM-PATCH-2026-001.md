# **Patch 3 — Grace–Anonymous Resolution (Two-Class Attestation Model)**

DeepSeek correctly identified a live tension: the Grace Protocol includes both an Anonymous Disclosure Channel (function two, original) and Faithful Receipt of Offering (function six, ASCENT extension). A truly anonymous offering has no identity anchor; without an identity anchor there is no custody chain; without a custody chain Faithful Receipt cannot attest. The worked example in ASCENT v1.1 used pseudonymous sources (FAVORS UUID, CBGMODD device IDs) and avoided rather than resolved the tension.

Resolution: a two-class attestation model. Offerings to the Grace Protocol are sorted at intake into one of two classes, each routed through a different sub-pipeline.

Class A — Attested Offering. Source bears a pseudonymous identity anchor (UUID, device ID, signed credential) sufficient to establish a custody chain. Faithful Receipt applies in full: source signed, timestamped, resonance cross-verified, custody chained under CCAL. Class A offerings flow into the Federated Reserve as attested INPUT and are eligible for Dictum reference. The pseudonymous identity protects the source from retaliation while preserving instrument-level verifiability.

Class B — Anonymous Disclosure. Source bears no identity anchor. Anonymous Disclosure Channel applies in full: the offering is received without source attribution and routed to a separate Anonymous Reserve. Anonymous Reserve offerings are not eligible for direct Dictum reference because the custody chain is missing. Instead, they serve as triggers for independent investigation: an anonymous disclosure prompts the relevant SSPS node to seek corroborating Class A offerings or to initiate independent attested sensing. If Class A corroboration is obtained, the matter proceeds through the standard pipeline; if not, the anonymous offering remains in the Anonymous Reserve as a flagged signal without authorizing disbursement.

This resolves the tension by acknowledging that anonymity-against-retaliation and instrument-level-attestation are different protections serving different purposes. Anonymity protects the source from human consequences; attestation protects the system from corrupted INPUT. Both are necessary; neither subsumes the other; the two-class model preserves both.

The pseudonymous protection used in the worked example (FAVORS UUID \+ CBGMODD device IDs) is now correctly classified as Class A — pseudonymous, attested, anti-retaliation-protected through the device ID's separation from operator identity. Truly anonymous offerings (a tip with no identifier of any kind) are now correctly routed to Class B with the corroboration-required protocol.

Insertion point: ASCENT v1.2 Logic Layer gains the two-class attestation model as a sub-section under the Grace Protocol extension. The worked example gains a one-paragraph note classifying the port-facility scenario as Class A. A new Class B worked example may be added in a future revision.
